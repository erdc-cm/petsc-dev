Future Plans for PETSc

(Some thoughts)

 Bill, Barry

  • 1) More complete support for defining and solving PDE problems: meshes, 
    discretizations, and problem definitions. This could start with more 
    complete support for DAs and move toward supporting unstructured grids.

    Pro: This would make a lot of people very happy.
    Con: Where is the research? Is this what we should be doing?
    Risk: Cai tells me that Farhad is developing a PDE solver around 
    FETI. Others are doing similar things. PETSc will always be better at 
    what it does, but more integrated environments may be easier for most 
    people to use. If we go this route, we're competing directly with other 
    groups.

We should do a small amount of work for DAs and concentrate our efforts on a unstructured grid infrastructure using the distributed data base idea.

  • 2) More support for interaction with other components. In other words, 
    work more aggressively to extend PETSc by making it easier to interoperate 
    with other tools. 

Pro: This would keep us ahead of the competition. Further, developing the 
abstractions and supporting tools to implement the interoperability would 
be interesting research. We could all work on TAO :)
Con: We'd have to work with and around the flaws in other tools. Where no 
tool was available, we'd be back at case 1.
Risk: This is hard. We could fail. Even if we succeeded, we'd have to 
pick other tools wisely. Politically, we'd have to do this by working 
within the CCA and/or ACTS.

We can make PETSc ESI and CCA compliant and then make interfaces to "fill in the gaps" in PETSc, for example, parallel direct solvers, eigenvalues/eigenvectors, ??? It is too difficult and nonproductive to try to interface with SAMRAI, Overture etc since those packages don't closely enough fit. 

  • 3) Identify and pursue a next-generation programming model. For example, 
    we might decide on remote-memory-access connected SMP clusters. We could 
    design in more dynamic data structures and load balancing/management.

    Pro: We'd get to write everything over again. If done right, it could be a 
    good fit to Playstation2 clusters and PIM-based systems.
    Con: We'd get to write everything over again.
    Risk: If we guess wrong on the programming model, we're toast.

I'm not so interested in this. I think current PETSc could be fine with Playstation 2.

  • 4) More of the same.

    Pro: We get to make PETSc increasingly feature-rich, robust, and usable.
    Con: Where is the research? How can we support it? How can MICS get 
    support for it?
    Risk: The field passes us by. Think PVM (which is still being make more 
    robust and usable, btw).
  • 5) Leverage the current PETSc by attacking more significant applications

    Pro: We'll make friends, learn about the unmet needs of computational 
    scientists, possibly leading to new research directions.
    Con: This is eating our seed corn.
    Risk: We get side-tracked into appliation tar pits. Also, current MICS 
    thinking is downplaying direct involvement in applications.

We need to pick an area that has not already been beaten to death and does not have a huge legacy of codes and ways of thinking about things, cause we'll never make an impact. So, no CFD, no structures, no global climate, no combustion,. We need a radically new area that we can make in our image. I'd like to do modeling the brain or something like that.

  • 6) Tackle completely different problem areas. For example, n-body computations

    Pro: Its different, its hard, and we would bring a different perspective to 
    the problem.
    Con: Its hard and many groups have/are developed n-body codes. We'd have 
    to have a clear advantage.
    Risk: We'd be entering the game late. We might not be able to catch up

 

  • 7) Start a new project.

    Pro: Could be *very* exciting.
    Con: We'd have to learn enough to enter the field. Its hard but not 
    impossible; consider Globus and security
    Risk: !!!!!!!!!!!!!!!

Is Globus and security an example of what to do, or what not to do? I'd like to stick to problems where PDEs are an issue
